# Phase 11 — Report Generation Audit: Context

**Purpose:** Guide the audit to verify that reports are built from stored atoms and lineage with zero introduction of new claims.

---

## 0. Locked Decisions (Non-Negotiable)

The following decisions are **final** and must be enforced in all deliverables:

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | **V3Retriever ONLY** for evidence gathering | Phase 10 truth contract; HybridRetriever is non-compliant |
| 2 | **Binary refusal for insufficient evidence** — do not write sections lacking atoms | Matches Phase 10 refusal semantics; prevents speculation |
| 3 | **Citation format: [A###]** sequentially per report | Reuse Phase 10 validator; ensure determinism |
| 4 | **Remove "minimum word count"** from synthesis prompts | Prevents hallucination pressure |
| 5 | **Store atom IDs per section** for regeneration provenance | Deterministic lineage; verifiable from Postgres |
| 6 | **mission_id is canonical** — bind all report artifacts to mission_id | Consistent with V3 identity model |
| 7 | **Contradictions must be explicitly stated** when relevant to section topic | Truth preservation; no silent resolution |
| 8 | **LLM-structured reports allowed** but structure must only organize, not synthesize | Flexibility in outline, but no new claims |
| 9 | **Report = pure transformation** — zero inference, no new facts | Core truth contract: representation only |

## 1. Truth Contract (Carry-Forward from Phase 10)

All reports must adhere to the grounding rules established in Phase 10:

- **Grounded answers only:** Every factual claim must map directly to a retrieved atom or be a trivial rephrasing.
- **No inference:** Cannot fill gaps, combine atoms to create new conclusions, or extrapolate.
- **Binary coverage:** All material claims must have at least one supporting citation; otherwise, the section must be marked `[INSUFFICIENT EVIDENCE]`.
- **Explicit refusal:** Sections without sufficient evidence are not written; the report must explicitly indicate insufficiency.
- **Mandatory retrieval:** All evidence must come from `V3Retriever`; no other data sources.
- **No hard filtering:** Return all relevant atoms regardless of confidence; filtering at retrieval stage is forbidden.
- **Contradictions preserved:** Conflicting atoms must appear in evidence and be acknowledged in prose when germane.
- **Eventual consistency:** Query operates on ChromaDB-indexed atoms only; missing due to indexing delay counts as insufficient evidence.
- **System prompt constraints:** LLM must be instructed: "Use ONLY the provided evidence... Do NOT use your general training... Every claim must be cited."

## 2. Report Generation Contract

### 2.1 Data Source

- **Allowed:** `V3Retriever.retrieve()` → atoms from `knowledge.knowledge_atoms` via Chroma
- **Forbidden:** `HybridRetriever`, `MemoryManager`, live web queries, any source outside V3 atom store

### 2.2 Section Writing Rule

For each section in the report plan:

1. **Evidence assembly:** Build evidence packet from `V3Retriever` using `mission_id` (not just topic_id)
2. **Sufficiency check:** If `len(atoms) < MIN_FOR_SECTION` (deterministic threshold, e.g., 3 atoms), **skip writing** and insert:

   ```markdown
   ## [Section Title]

   [INSUFFICIENT EVIDENCE FOR SECTION]
   ```

3. **Writing:** Only if sufficient evidence, call `ArchivistSynthAdapter.write_section(packet)` with prompt that:
   - Removes any word count minimum
   - Requires `[A###]` citations inline
   - Requires acknowledgment of contradictions if present in packet
   - Forbids use of internal knowledge

### 2.3 Citation Assignment

- During `build_evidence_packet()`, assign sequential `citation_key` to each retrieved atom (e.g., `[A001]`, `[A002]`).
- The `global_id` field in the packet must carry this citation key.
- Archivist must copy these keys verbatim into the prose.

### 2.4 Provenance Recording

For each section that is written, store in the synthesis artifact:

```json
{
  "section_title": "...",
  "section_order": N,
  "atom_ids_used": ["atom_id_1", "atom_id_2", ...]  // ordered as cited
}
```

This enables verification: every claim can be traced back to a specific atom in Postgres.

### 2.5 Contradiction Handling

- `EvidenceAssembler` should include contradictions in the packet if topic includes "contradictions" in target roles OR if contradictions exist among the retrieved atoms.
- When writing the section, if contradictions are present, Archivist must include a statement like:

  ```markdown
  Sources conflict on this point. [A003] states X, while [A007] states Y.
  ```

- Do **not** resolve or pick a side; just surface.

### 2.6 Report Structure

- The section plan may be generated by LLM (as currently done) **provided** the plan only organizes evidence areas and does not introduce new claims.
- Deterministic templates are acceptable but not required.

## 3. Verification Criteria (Phase 11 PASS Requirements)

Phase 11 passes verification only if **all** are true:

1. **Evidence path:** All atoms used in reports come from `V3Retriever` (no HybridRetriever usage)
2. **Lineage traceability:** Every sentence in a written section can be mapped to at least one atom ID from the stored `atom_ids_used` list
3. **Insufficiency handling:** Sections lacking minimum evidence produce `[INSUFFICIENT EVIDENCE FOR SECTION]`, not speculative prose
4. **Citation integrity:** All inline citations match `[A###]` format and correspond to atoms in `atom_ids_used`
5. **Mission binding:** Report artifact metadata includes `mission_id`; all queries filtered by `mission_id`
6. **Regeneration possible:** Given `mission_id` and the stored `atom_ids_used` lists, the exact same report can be regenerated (deterministic LLM sampling disabled or fixed seed)
7. **Contradictions surfaced:** When contradictory atoms are in the packet, the prose explicitly mentions the disagreement
8. **No hallucination pressure:** Archivist prompt does **not** include word count minimums or "fill gaps" language

**Hard fail conditions:**
- Report content includes facts not present in the cited atoms
- Any code path bypasses V3Retriever
- `atom_ids_used` not stored or incomplete
- Evidence insufficient but section still written with claims
- Contradictions silently ignored
