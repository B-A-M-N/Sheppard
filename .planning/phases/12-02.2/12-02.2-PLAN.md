# Phase 12-02.2: Batch Multi-Query Retrieval — Execution Plan

**Objective:** Reduce total retrieval latency to ≤200–300ms by batching section queries into a single ChromaDB call.

## Tasks

### Task 1: Add `V3Retriever.retrieve_many` for batched queries

Implement new method:

```python
async def retrieve_many(self, queries: List[RetrievalQuery]) -> List[RoleBasedContext]:
    """
    Retrieves context for multiple queries in a single Chroma query.
    All queries must share the same where filter (mission_id or topic_id).
    Returns list of RoleBasedContext in same order as queries.
    """
```

Implementation:
- Validate all queries have same filter type and value (enforce at call site; warn if not).
- Build common `where` dict from first query.
- Call `self.adapter.chroma.query(collection="knowledge_atoms", query_texts=[q.text for q in queries], where=where, limit=queries[0].max_results)`.
- The result will have `documents[0]`, `metadatas[0]`, `distances[0]` as lists of length N (one per query). Need to split into per-query slices.
- For each query index, build a `RoleBasedContext` with `evidence` items from that slice, identical to the loop in `retrieve()`.
- Preserve `_profile` if `PROFILE_RETRIEVAL` is set: compute per-query times (post, query) as in `retrieve`. For batched call, the total time includes all queries; we can estimate each as (total / N) or record single-pass overhead only. Simpler: for each context, set `_profile = {"query_ms": total_ms / len(queries), "post_ms": 0, "total_ms": total_ms / len(queries)}`. This is approximate but acceptable for efficiency measurement. Optionally, we could pre-and post-time each slice separately but that requires iterating twice. Batch goal is to report single concurrent total; earlier benchmark computes sum of individual times from `build_evidence_packet` patching, which is independent. We can leave profiling off in batched path, or attach dummy uniform values.

**Acceptance:** `retrieve_many` returns `List[RoleBasedContext]` with same content as successive `retrieve` calls (for same queries) and respects where filtering. Test manually with 3 queries.
