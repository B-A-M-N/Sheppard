# Phase 12-02.1: Retrieval Latency Diagnosis — Execution Plan

**Objective:** Profile and measure the components of retrieval latency to identify the bottleneck causing ~1s per-query cost.

## Tasks

### Task 1: Add detailed profiling instrumentation to `EvidenceAssembler.build_evidence_packet`

Add timing splits to measure:
- Embedding generation time (if applicable in `retriever.retrieve`)
- ChromaDB query execution time (inside `retriever.retrieve`)
- Metadata filtering/post-processing time

Integrate with existing `RETRIEVAL_CONCURRENCY_LIMIT` and debug logging to output:
```
[Assembler] Section '{title}' — embedding: Xms, query: Yms, post: Zms, total: Tms
```

**Files to modify:**
- `src/research/reasoning/v3_retriever.py` (add timing around `self.chroma.query` and embedding calls)
- `src/research/reasoning/assembler.py` (add post-retrieval processing time)

**Acceptance:** When running benchmark, logs show all three components with non-zero values.

### Task 2: Create diagnostic benchmark script `scripts/diagnose_retrieval_latency.py`

This script will:
- Reuse `EvidenceAssembler` production path
- Support `--corpus-tier` (small/medium/large)
- Support `--db-state` (clean vs current):
  - clean: create fresh Chroma DB at temp path, seed with specified tier only
  - current: use existing DB
- Support `--queries` (1 or 8) to measure single-query vs concurrent
- Output structured JSON with:
  - corpus_tier, corpus_atom_count, db_state, concurrent_queries
  - per_section_timings list (embedding, query, post, total)
  - aggregated means and totals

**Acceptance:** Script runs without errors and produces JSON with full timing breakdown.

### Task 3: Run diagnostic matrix and collect data

Execute the following combinations (2 DB states × 3 tiers × 2 query counts = 12 runs):
```bash
python scripts/diagnose_retrieval latency.py --scenario high_evidence --corpus-tier small --db-state clean --queries 1
python scripts/diagnose_retrieval latency.py --scenario high_evidence --corpus-tier small --db-state clean --queries 8
python scripts/diagnose_retrieval latency.py --scenario high_evidence --corpus-tier medium --db-state clean --queries 1
python scripts/diagnose_retrieval latency.py --scenario high_evidence --corpus-tier medium --db-state clean --queries 8
python scripts/diagnose_retrieval latency.py --scenario high_evidence --corpus-tier large --db-state clean --queries 1
python scripts/diagnose_retrieval latency.py --scenario high_evidence --corpus-tier large --db-state clean --queries 8
# Repeat for --db-state current
```

Store results in `.planning/phases/12-02.1/diagnostic_results/` as separate JSON files.

**Acceptance:** All 12 runs complete; results directory populated.

### Task 4: Analyze results and write report

Create `.planning/phases/12-02.1/ANALYSIS.md` summarizing:
- Which component dominates latency (embedding, query, post)?
- How does latency scale with corpus size? Linear? Super-linear?
- Difference between clean and current DB (bloat evidence)?
- Single-query vs concurrent: any overhead? Efficiency?
- Mission_id filtering impact (inferred from post time)
- Recommendations: Is per-query latency fixable? If so, how?

**Acceptance:** ANALYSIS.md provides data-driven answers to the 6 diagnosis questions and a recommendation (fix & re-run 12-02 vs adjust target).

## Guardrails

- Do not modify production code in ways that change semantics; only add timing instrumentation
- Preserve existing guardrail tests (99 tests must still pass)
- Ensure instrumentation adds negligible overhead (<1%)

## Success Criteria

Phase is successful if it produces a clear diagnosis with actionable next steps for either fixing the bottleneck or revising PERF-01.
