# Context

Phase 12-02 implemented concurrent retrieval via `EvidenceAssembler.assemble_all_sections`. Concurrency works (6.43× efficiency), but absolute retrieval latency remains ~1066ms total, with per-query ~980ms. The PERF-01 target (≤200–300ms total) assumes per-query latency of ~150ms.

The verifier concluded: "per-query latency higher than baseline; may be due to corpus bloat or indexing changes." We need to diagnose before adjusting targets or proceeding to Phase 12-03.

# Diagnosis Goal

Answer:
1. Is time dominated by embedding generation, Chroma query, or metadata filtering?
2. Is benchmark running against a bloated/dirty corpus?
3. Does a fresh DB materially reduce latency?
4. Does corpus size scale linearly/sublinearly/pathologically?
5. Does mission_id filtering contribute meaningfully to cost?
6. Is there unnecessary repeated work per section query?

# What We Will Test

Minimal matrix:
- fresh DB vs current DB
- small / medium / large corpus
- embedding time vs vector query time split
- cold vs warm cache (where applicable)
- one query vs eight-query concurrent run
- same mission with/without accumulated atoms

# What We Will NOT Change

- Truth invariants
- Heuristic shortcuts
- Persistent caching strategies
- PERF-01 target (diagnosis only; decision after findings)

# Decision Rule After Diagnosis

- If fixable backend pathology identified, fix and re-run Phase 12-02 benchmark
- If target unrealistic even on clean corpus, revise PERF-01 with evidence
- Only then proceed to Phase 12-03
