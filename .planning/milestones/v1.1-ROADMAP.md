# Milestone v1.1: Performance & Observability

**Status:** ✅ SHIPPED 2026-03-31
**Phases:** 12-01–12-07 (includes 12-02.1, 12-02.2)
**Total Plans:** 13

## Overview

v1.1 optimized the truth-safe system built in v1.0 by adding performance monitoring, distributed tracing, V3-native contradiction handling, high-evidence E2E validation, and constraint-safe ranking. All truth contract invariants preserved.

Key deliverables:
- Benchmark suite with baseline metrics (12-01)
- Retrieval latency reduced from ~1200ms to ≤300ms via batch queries (12-02, 12-02.1, 12-02.2)
- Structured JSON logging and mission-scoped tracing (12-04)
- Contradictions upgraded to V3-native with atom attribution (12-05)
- Full E2E integration test with real atoms (12-06)
- Deterministic ranking with composite scoring, no filtering (12-07)

Known limitation: Synthesis throughput (PERF-02) not improved due to single-endpoint inference serialization. Architecture correct; deployment-bound. Mitigated via `SYNTHESIS_CONCURRENCY_LIMIT=1`.

### Phase 12-02.1: Retrieval Latency Diagnosis (INSERTED)

**Goal:** Identify root cause of high per-query latency observed after 12-02 implementation.
**Status:** ✅ Completed (Diagnostic)
**Artifacts:** ANALYSIS.md with full findings and recommendations

**Details:**
Profiling showed that individual queries are fast (~150–180ms), but concurrent retrieval (8 sections) suffers severe serialization due to GIL contention during embedding computation. Recommended fix: batch section queries into a single Chroma call to eliminate per-query overhead.

## Milestone Summary

### Decimal Phases

- Phase 12-02.1: Retrieval Latency Diagnosis (inserted after 12-02 to diagnose throughput regression)
- Phase 12-02.2: Batch Multi-Query Retrieval (inserted after 12-02.1 to implement fix)

### Key Decisions

- Use mission_id as trace ID (simplicity, no separate trace_id)
- JSON logs only (no Prometheus/OpenTelemetry dependencies)
- Bounded async worker pool for synthesis (correct architecture, even if single-endpoint limits speed)
- Ranking opt-in via `enable_ranking` flag (backward compatible)
- Keep default lexical ordering when ranking disabled (exact byte-for-byte compatibility)

### Issues Resolved

- Retrieval latency too high after initial parallelization (GIL contention) → diagnosed and fixed with batch queries
- Legacy contradiction retrieval dependency → fully V3-native
- Missing high-evidence E2E coverage → created verifier and passed
- Ranking design (constraint-safe implementation) → implemented and fully tested

### Issues Deferred

- PERF-02 (synthesis throughput ≥20% improvement): blocked by single-endpoint inference serialization. Architecture ready; requires multi-endpoint deployment or batch inference API. Documented in `.planning/phases/v1.1/MILESTONE-RECONCILIATION.md`.

### Technical Debt Incurred

- None. All implementations follow guardrails; no shortcuts taken.
