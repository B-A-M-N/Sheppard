# Phase 03.0 — Canonical Authority Lock — Context

**Purpose**: Guide implementation of Phase 03.0 (precedes Triad Enforcement)

---

## Decided Architecture

- **Single Canonical Truth**: Postgres (V3) only
- **V2 Deprecation**: `MemoryManager` and `semantic_memory` DB are legacy, not runtime
- **No Fallbacks**: V3 operations fail explicitly if canonical data missing
- **Retriever**: `V3Retriever` only (not `HybridRetriever`)

---

## Implementation Boundaries

### Must Do

- Remove `MemoryManager` instantiation from `SystemManager.initialize`
- Remove all imports of `src.memory.manager` in V3 code (`src/core/`, `src/research/`)
- Delete or relocate `HybridRetriever` class (outside `src/research/reasoning/`)
- Wire `V3Retriever(adapter=self.adapter)` in `SystemManager`
- Add defensive guard in `MemoryManager.__init__` to raise if instantiated
- Verify system boots with only V3 adapter
- Update `ARCHITECTURE.md` to reflect V3 canonical authority

### Must Not Do

- Add compatibility shim that reads V2 during V3 operation
- Implement fallback logic (V3 → V2) under any condition
- Keep `HybridRetriever` alive in V3 path
- Modify V2 data as part of normal V3 operation
- Defer removal with "temporary" flags

---

## Verification Criteria (Binary)

- [ ] `SystemManager` has no `memory` attribute
- [ ] No `from src.memory.manager import MemoryManager` in V3 runtime modules
- [ ] `HybridRetriever` not importable from `src.research.reasoning`
- [ ] `/query` executes without touching `semantic_memory` DB
- [ ] Application start-up completes with only V3 adapter

---

## Known Risks

- **Initial emptiness**: After lock, V3 knowledge may be empty → queries return nothing. That is acceptable. The system must populate V3 via `/learn` pipeline.
- **V2 data abandonment**: Existing `semantic_memory` data will not be accessible. Migration is out-of-scope (optional future script).
